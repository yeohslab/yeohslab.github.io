# 本仓库文档（Fork 维护版）

本文档用于说明：当你基于开源博客项目 Fork 并改造成个人站时，如何在不违反社区规则的前提下持续维护。

## 一、你需要遵守的 Fork 基本规则

- 保留上游许可证文件（如 `LICENSE`）。
- 保留必要的版权声明，不擅自移除源码中的许可证注释。
- 在文档中明确“本项目基于哪个上游仓库 Fork”。
- 对外说明你的新增改动，避免与上游贡献混淆。
- 回馈通用修复（可选但推荐）：通过 PR 向上游贡献。

## 二、建议保留的仓库说明结构

建议在仓库根目录或文档中长期维护以下信息：

1. 上游来源（项目链接）
2. 本仓库定位（个人博客 / 二次开发版本）
3. 本仓库差异（你做了哪些改动）
4. 本地运行方式
5. 部署方式（GitHub Pages）

## 三、当前项目的改造建议

### 1) 站点身份改造

请在 `_config.yml` 中维护：

- `title`
- `SEOTitle`
- `description`
- `keyword`
- `url`
- `baseurl`

### 2) 第三方服务改造

- `disqus_username`：不用就留空
- `ga_track_id`：不用就注释
- `ga_domain`：一般用真实域名或 `auto`

### 3) 内容改造

- 替换 `about` 页面
- 清理不相关历史文章
- 添加你自己的首批文章与标签体系

## 四、社区友好的提交方式（推荐）

- 每次提交只做一类变更（配置 / 样式 / 功能 / 文档分开）
- 提交信息说明“为什么改”，不只写“改了什么”
- 示例：
  - `docs: rewrite fork rules for this repo`
  - `feat: support xiaohongshu and kuaishou links`
  - `fix: center social icons in sidebar`

## 五、与上游同步（可选但推荐）

如果你希望继续获得上游修复，建议：

1. 配置上游远端 `upstream`
2. 定期同步并在分支处理冲突
3. 优先保留你的站点配置（如 `_config.yml`）
4. 同步完成后本地预览再合并

## 六、发布前自检

- [ ] 许可证文件仍在
- [ ] 已注明上游来源
- [ ] 无原作者账号残留在配置中
- [ ] 评论/统计配置符合你的实际需求
- [ ] 页面链接、图标、头像显示正常
- [ ] 文档与当前代码行为一致

## 七、免责声明建议（可选）

你可以在 README 增加类似说明：

> 本项目基于开源主题 Fork 并进行二次开发。  
> 主题版权与许可归原作者及其许可证所有；本仓库仅包含个人化配置与功能改造。
