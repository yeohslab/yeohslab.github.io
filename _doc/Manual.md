# Fork 项目维护手册（适配本仓库）

> 本文档面向从开源博客模板 Fork 后进行个人化改造与长期维护的场景。
> 目标是：合规、可维护、可持续更新上游。

## 1. 开源合规原则（必须）

1. 保留上游许可证与版权声明  
   - 不删除仓库中的 `LICENSE`。  
   - 不删除源码中已有的许可证头注释（如有）。

2. 保留来源说明（Attribution）  
   - 在 README 或文档中注明本项目基于哪个上游仓库 Fork。  
   - 对外发布时，避免让读者误以为主题完全原创。

3. 标注你的修改范围  
   - 建议在文档加入“本仓库改动”章节，说明你改了哪些功能、样式、配置。  
   - 避免把上游功能变更误写成你原创实现。

4. 遵守上游许可证条款  
   - 若上游是 Apache-2.0 / MIT / GPL，请按各自要求保留声明与分发条款。  
   - 商用、二次分发前先确认许可证兼容性。

## 2. Fork 后第一步改造清单

优先修改 `_config.yml`，保证站点身份与配置独立于原作者：

- 站点身份：`title`、`SEOTitle`、`description`、`keyword`
- 站点地址：`url`、`baseurl`
- 账号信息：`github_username` 等社交字段
- 头像与简介：`sidebar-avatar`、`sidebar-about-description`
- 评论与统计：`disqus_username`、`ga_track_id`（不用就留空或注释）
- 时区与发布：`timezone`、`future`

同时建议处理：

- `CNAME`：无自定义域名时删除
- `_posts/`：移除与自己无关的历史文章
- `about.html` 与 `_includes/about/zh.md`：替换为个人介绍

## 3. 分支与提交规范（推荐）

建议使用以下工作流：

1. `main` 保持可部署状态  
2. 新功能/修复使用短分支（如 `feat/social-icons`）  
3. 每次提交聚焦单一目的，提交信息清晰可追溯  
4. 合并前本地预览通过（`bundle exec jekyll serve`）

提交信息建议：

- `docs: rewrite fork maintenance guide`
- `feat: add xiaohongshu and kuaishou social links`
- `fix: center custom social icons in sidebar`

## 4. 与上游同步策略（长期维护重点）

建议保留 `upstream` 远端，周期性同步：

1. 添加上游远端（仅首次）
2. 定期 `fetch upstream`
3. 在独立分支处理冲突
4. 验证页面后再合并到 `main`

冲突处理原则：

- 配置文件（如 `_config.yml`）以你的站点配置为准
- 模板结构变更优先吸收上游修复，再回放你的个性化改动
- 每次同步写一条简短变更记录

## 5. 文档与透明度（社区友好）

建议在仓库保持以下文档：

- `README.md`：项目说明、预览地址、快速启动
- `_doc/README.zh.md`：中文维护说明（面向自己和协作者）
- `CHANGELOG`（可选）：记录显著变更

对外协作时建议写明：

- 哪些功能来自上游
- 哪些功能是本仓库新增
- 如何复现本地开发环境

## 6. 安全与隐私

- 不提交密钥、令牌、隐私配置（邮箱可按需公开）
- 统计、评论等第三方服务默认最小化启用
- 外链资源（图标/CDN）要考虑可用性与隐私影响

## 7. 发布前检查清单

- [ ] `_config.yml` 中无原作者账号残留
- [ ] `url/baseurl` 与仓库部署方式一致
- [ ] 侧边栏、社交链接、头像显示正常
- [ ] 评论/统计为你自己的配置，或已关闭
- [ ] 首页与归档页无无关历史内容
- [ ] 文档中保留了上游来源与许可证说明

## 8. 常见问题（FAQ）

### Q1：Fork 后可以删除原作者信息吗？

可以做品牌替换，但不能违反许可证要求。应保留必要的版权/许可证声明，并在文档注明来源。

### Q2：只改了样式，算不算衍生作品？

通常仍属于在开源项目基础上的衍生修改，建议保留来源与许可证。

### Q3：我新增了很多功能，要不要回馈上游？

建议把通用改进（bugfix、兼容性修复）以 PR 回馈上游，这也是开源社区常见协作方式。
